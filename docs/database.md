# PIA Tracker — Database

**Status:** Draft v1.
**See also:** `architecture.md` § 4 (core patterns), § 5 (data model overview).

This document is the definitive reference for the database schema, indexing strategy, JSONB conventions, partitioning, and migration discipline. Every column listed here should appear in a Flyway migration; every JSONB content shape should be referenced by a `form_definitions` or `workflow_definitions` row.

---

## 1. Conventions

**Engine.** PostgreSQL 16+. No multi-DB support; Postgres-only features used freely (JSONB, GIN indexes, partitioning, triggers, generated columns).

**Identifiers.** UUID v4, generated by Postgres `gen_random_uuid()`. Column type `uuid`. No surrogate `bigint` IDs.

**Timestamps.** All time columns are `timestamptz` (stored UTC). The audit columns `created_at`, `updated_at` exist on every domain table; `created_at` defaults to `now()`, `updated_at` is maintained by a trigger.

**Naming.**

| Object | Convention |
|---|---|
| Tables | `snake_case`, plural noun (`projects`, `activity_records`) |
| Columns | `snake_case` |
| Primary key | `id` always |
| Foreign keys | `{referenced_table_singular}_id` (e.g., `project_id`) |
| JSONB columns | `{purpose}_json` (e.g., `schema_json`, `metadata_json`) |
| Audit columns | `created_at`, `created_by_user_id`, `updated_at`, `updated_by_user_id` |
| Soft-delete columns | `is_deleted boolean default false`, `deleted_at`, `deleted_by_user_id` |
| Indexes | `ix_{table}_{column[_column]}` for btree, `gin_{table}_{column}` for GIN |
| Constraints | `chk_{table}_{purpose}`, `uq_{table}_{column[_column]}` |

**Soft delete behavior.** Domain repositories filter `is_deleted = false` by default via Hibernate `@SQLRestriction`. Explicit `findIncludingDeleted` variants exist where needed. Audit log records the soft delete.

**Optimistic locking.** Every domain entity has `version integer not null default 0`, mapped via Hibernate `@Version`. Surfaced to the API as an ETag header.

**Money and area.** `numeric(15, 4)` for hectares (4 decimal places). `numeric(15, 2)` for money. Never `float` or `double precision`.

## 2. Structural tables — zones, divisions, designations, users

### `zones`

```sql
create table zones (
    id              uuid primary key default gen_random_uuid(),
    code            varchar(16) not null unique,    -- e.g., 'NR', 'WR', 'CR'
    name            varchar(128) not null,           -- e.g., 'Northern Railway'
    short_name      varchar(32) not null,
    display_order   integer not null default 0,
    is_active       boolean not null default true,
    created_at      timestamptz not null default now(),
    updated_at      timestamptz not null default now()
);
```

Seeded by Flyway from the 17 zones of Indian Railways.

### `divisions`

```sql
create table divisions (
    id              uuid primary key default gen_random_uuid(),
    zone_id         uuid not null references zones(id),
    code            varchar(16) not null,            -- e.g., 'DLI', 'BCT'
    name            varchar(128) not null,
    display_order   integer not null default 0,
    is_active       boolean not null default true,
    created_at      timestamptz not null default now(),
    updated_at      timestamptz not null default now(),
    unique (zone_id, code)
);

create index ix_divisions_zone_id on divisions(zone_id);
```

Seeded by Flyway.

### `designations`

```sql
create table designations (
    code                  varchar(32) primary key,    -- e.g., 'SR_DEN', 'DY_CSTE'
    name                  varchar(128) not null,       -- 'Senior Divisional Engineer'
    short_label           varchar(32) not null,        -- 'Sr DEN'
    category              varchar(32) not null,        -- CONSTRUCTION/ELECTRICAL/STE/OPERATIONS/...
    is_approval_role      boolean not null default false,
    is_data_entry_role    boolean not null default false,
    display_order         integer not null default 0,
    description           text,
    created_at            timestamptz not null default now(),
    constraint chk_designations_at_least_one_role check (is_approval_role or is_data_entry_role)
);
```

The full registry (~25 designations) is seeded by Flyway from the requirements documents. New designations are added by data migration, not code. `SR_DEN` and `SR_DEN_CO` are distinct rows (decision IIII). `is_approval_role` is true for designations that can appear on a drawing's approver list; `is_data_entry_role` is true for designations that can hold project assignments and create records. A designation can be both.

### `users`

```sql
create table users (
    id                       uuid primary key default gen_random_uuid(),
    employee_id              varchar(32) unique,         -- Railway HR ID (null for system users)
    name                     varchar(256) not null,
    email                    varchar(256) not null unique,
    designation_code         varchar(32) not null references designations(code),
    primary_zone_id          uuid references zones(id),  -- null for HQ users
    primary_division_id      uuid references divisions(id),
    is_active                boolean not null default true,
    is_system_user           boolean not null default false,
    last_login_at            timestamptz,
    -- audit + soft delete
    created_at               timestamptz not null default now(),
    created_by_user_id       uuid references users(id),
    updated_at               timestamptz not null default now(),
    updated_by_user_id       uuid references users(id),
    is_deleted               boolean not null default false,
    deleted_at               timestamptz,
    deleted_by_user_id       uuid references users(id),
    version                  integer not null default 0
);

create index ix_users_designation_zone on users(designation_code, primary_zone_id)
    where is_active and not is_deleted;
create index ix_users_email on users(email);
```

The composite `(designation_code, primary_zone_id)` index is the hot path: every user picker filters by both.

### `user_zone_assignments` — cross-zone access

```sql
create table user_zone_assignments (
    id                       uuid primary key default gen_random_uuid(),
    user_id                  uuid not null references users(id),
    zone_id                  uuid not null references zones(id),
    granted_by_user_id       uuid not null references users(id),
    granted_at               timestamptz not null default now(),
    expires_at               timestamptz,             -- null = permanent
    reason                   text,
    is_active                boolean not null default true,
    unique (user_id, zone_id)
);

create index ix_uza_user_id on user_zone_assignments(user_id) where is_active;
```

Picker queries union the user's `primary_zone_id` with their active, unexpired `user_zone_assignments`.

## 3. Authorization tables — permissions, roles, system grants

### `permissions`

```sql
create table permissions (
    code           varchar(128) primary key,    -- 'PROJECT.READ.ZONE', 'ACTIVITY_RECORD.AUTHENTICATE'
    description    text not null,
    category       varchar(64) not null,         -- PROJECT/ACTIVITY/RECORD/FORM/DASHBOARD/EXPORT/USER/SYSTEM
    is_system_grant boolean not null default false  -- true for permissions like VIEW_PAN_INDIA
);
```

Seeded by Flyway. The full permission catalog is in `permissions.md`. Adding a permission is a Flyway data migration — never a code constant.

### `roles`

```sql
create table roles (
    code           varchar(64) primary key,     -- 'CE_C_CONSTRUCTION', 'NODAL_DY_CE_C', 'ADMIN'
    name           varchar(128) not null,
    description    text,
    is_active      boolean not null default true,
    created_at     timestamptz not null default now()
);

create table role_permissions (
    role_code        varchar(64) not null references roles(code),
    permission_code  varchar(128) not null references permissions(code),
    primary key (role_code, permission_code)
);
```

Roles bundle permissions; users get permissions through role membership and through ad-hoc grants.

### `designation_default_roles`

```sql
create table designation_default_roles (
    designation_code  varchar(32) not null references designations(code),
    role_code         varchar(64) not null references roles(code),
    primary key (designation_code, role_code)
);
```

When a user is created, their designation determines which roles they get by default. Admin can later modify.

### `user_roles` — actual role memberships

```sql
create table user_roles (
    user_id      uuid not null references users(id),
    role_code    varchar(64) not null references roles(code),
    granted_by_user_id uuid references users(id),
    granted_at   timestamptz not null default now(),
    primary key (user_id, role_code)
);
```

### `user_permissions` — ad-hoc grants

```sql
create table user_permissions (
    user_id            uuid not null references users(id),
    permission_code    varchar(128) not null references permissions(code),
    granted_by_user_id uuid not null references users(id),
    granted_at         timestamptz not null default now(),
    expires_at         timestamptz,
    reason             text,
    primary key (user_id, permission_code)
);
```

This is where `VIEW_PAN_INDIA` and other system grants live for individual users (decision I).

## 4. Project entity

### `projects`

```sql
create table projects (
    id                          uuid primary key default gen_random_uuid(),
    project_code                varchar(64) not null unique,    -- e.g., 'NR-NL-2024-017'
    name                        varchar(256) not null,
    project_type                varchar(64) not null,            -- NEW_LINE/DOUBLING/GAUGE_CONVERSION/ELECTRIFICATION/...
    zone_id                     uuid not null references zones(id),
    division_id                 uuid references divisions(id),   -- can be null until allocated
    chainage_from_km            numeric(10, 3),
    chainage_to_km              numeric(10, 3),
    length_km                   numeric(10, 3),
    recommended_by_board_on     date,                            -- powers "days since RB recommendation"
    target_completion_year      integer,
    lifecycle_state             varchar(32) not null,            -- DRAFT/AWAITING_CAO_ALLOCATION/AWAITING_CEC_ASSIGNMENT/ACTIVE/ON_HOLD/COMPLETED/DROPPED
    -- flexible metadata
    metadata_json               jsonb not null default '{}'::jsonb,
    -- audit + soft delete + lock
    created_at                  timestamptz not null default now(),
    created_by_user_id          uuid not null references users(id),
    updated_at                  timestamptz not null default now(),
    updated_by_user_id          uuid references users(id),
    is_deleted                  boolean not null default false,
    deleted_at                  timestamptz,
    deleted_by_user_id          uuid references users(id),
    version                     integer not null default 0
);

create index ix_projects_zone_state on projects(zone_id, lifecycle_state) where not is_deleted;
create index ix_projects_division_state on projects(division_id, lifecycle_state) where not is_deleted;
create index ix_projects_lifecycle on projects(lifecycle_state) where not is_deleted;
create index gin_projects_metadata on projects using gin (metadata_json);
```

`metadata_json` holds sanction details, cost estimates, scope notes, references to RB documents — anything that doesn't need a real column.

### `project_assignments`

```sql
create table project_assignments (
    id                  uuid primary key default gen_random_uuid(),
    project_id          uuid not null references projects(id),
    user_id             uuid not null references users(id),
    assignment_role     varchar(32) not null,        -- CAO_C/CE_C/DY_CE_C/NODAL_DY_CE_C
    assigned_by_user_id uuid not null references users(id),
    assigned_at         timestamptz not null default now(),
    is_active           boolean not null default true,
    deactivated_at      timestamptz,
    unique (project_id, user_id, assignment_role)
);

create index ix_pa_project_role on project_assignments(project_id, assignment_role) where is_active;
create index ix_pa_user on project_assignments(user_id) where is_active;
```

A user can hold multiple roles on the same project (e.g., NODAL_DY_CE_C is *in addition to* DY_CE_C — the Nodal row exists alongside the DY_CE_C row).

A project can have multiple activities of the same type (decision YYY — "Phase 1 LA" and "Phase 2 LA" are separate rows). Records reference the activity, not the project directly.

### `activity_records`

```sql
create table activity_records (
    id                       uuid primary key default gen_random_uuid(),
    project_activity_id      uuid not null references project_activities(id),
    -- Per-record overrides; default to the activity's defaults if null
    form_definition_id       uuid not null references form_definitions(id),
    workflow_definition_id   uuid references workflow_definitions(id),  -- null for drawings (uses checklist)
    -- The actual data
    data_json                jsonb not null default '{}'::jsonb,
    schema_version_at_save   integer not null,             -- denormalized form_def version snapshot
    -- Record-level status (derived from workflow_instances; cached here for fast list queries)
    record_state             varchar(32) not null,         -- DRAFT/IN_PROGRESS/SUBMITTED/VERIFIED/AUTHENTICATED/COMPLETE/SENT_BACK
    -- Discriminator for record type within the activity (e.g., 'ESP'/'SIP'/'GAD_MEGA' for drawings)
    record_subtype           varchar(64),
    -- audit + soft delete + lock
    created_at               timestamptz not null default now(),
    created_by_user_id       uuid not null references users(id),
    updated_at               timestamptz not null default now(),
    updated_by_user_id       uuid references users(id),
    is_deleted               boolean not null default false,
    deleted_at               timestamptz,
    deleted_by_user_id       uuid references users(id),
    version                  integer not null default 0
);

create index ix_ar_activity on activity_records(project_activity_id) where not is_deleted;
create index ix_ar_state on activity_records(record_state) where not is_deleted;
create index ix_ar_subtype on activity_records(record_subtype) where not is_deleted;
create index gin_ar_data on activity_records using gin (data_json);
```

For Land Acquisition, each record is one village. For Drawing Approval, each record is one drawing (with `record_subtype` = ESP / SIP / GAD_MEGA / etc.). For Utility Shifting, each record is one utility item (with `record_subtype` = LT_HT_EHV / PIPELINE / SNT / etc.).

## 6. Configuration tables — forms, workflows, dashboards

### `form_definitions`

```sql
create table form_definitions (
    id                    uuid primary key default gen_random_uuid(),
    activity_type_code    varchar(64) not null references activity_types(code),
    code                  varchar(64) not null,        -- e.g., 'LAND_ACQUISITION_V1'
    version               integer not null,
    label                 varchar(256) not null,
    schema_json           jsonb not null,              -- JSON Schema Draft 2020-12
    ui_schema_json        jsonb not null default '{}'::jsonb,  -- RJSF ui hints
    workflow_definition_id uuid references workflow_definitions(id),
    section_codes         text[] not null default '{}',  -- e.g., {SRP,CALA,SECTION_20A,JMR,...}
    default_approver_designations text[] not null default '{}',  -- drawings only
```

`section_codes` is the ordered list of section codes for forms that use section-level workflow (Land Acquisition, Forest Clearance). For others it's empty. `default_approver_designations` applies only to drawing form definitions and seeds the approver checklist at record creation.

## 7. Drawing approvers (separate from workflow engine)

```sql
create table drawing_approvers (
    id                            uuid primary key default gen_random_uuid(),
    activity_record_id            uuid not null references activity_records(id),
    approval_designation_code     varchar(32) not null references designations(code),
    user_id                       uuid references users(id),    -- null if slot not yet filled
    status                        varchar(16) not null default 'PENDING',  -- PENDING/APPROVED/SENT_BACK
    position                      integer not null default 0,   -- display order, not enforcement
    acted_at                      timestamptz,
    comment                       text,
    is_deleted                    boolean not null default false,
    created_at                    timestamptz not null default now(),
    updated_at                    timestamptz not null default now(),
    constraint chk_da_status check (status in ('PENDING','APPROVED','SENT_BACK'))
);

create index ix_da_record on drawing_approvers(activity_record_id) where not is_deleted;
create index ix_da_user_pending on drawing_approvers(user_id, status)
    where not is_deleted and status = 'PENDING';
```

The drawing's overall state is **derived** from these rows: if any row is `SENT_BACK`, drawing is `SENT_BACK`; else if any row is `PENDING`, drawing is `IN_APPROVAL`; else drawing is `APPROVED`. This computation lives in a service method; `activity_records.record_state` caches the result for fast list queries.

Decision AAAA: editable by Admin / project CE/C / Nodal Dy CE/C.
Decision BBBB: editing preserves existing APPROVED statuses.
Decision CCCC: send-back doesn't reset other approvers.
Decision DDDD: any approval-role can be added.

### `attachments`

```sql
create table attachments (
    id                       uuid primary key default gen_random_uuid(),
    entity_type              varchar(32) not null,
    entity_id                uuid not null,
    field_path               varchar(256),                -- JSONB path the attachment belongs to (e.g., 'srp.gazette_pdf')
    file_name                varchar(512) not null,
    content_type             varchar(128) not null,
    size_bytes               bigint not null,
    minio_object_key         varchar(512) not null,
    sha256_hex               varchar(64) not null,
    clamav_scan_result       varchar(32) not null,        -- CLEAN/INFECTED/ERROR
    clamav_scanned_at        timestamptz,
    uploaded_by_user_id      uuid not null references users(id),
    created_at               timestamptz not null default now(),
    is_deleted               boolean not null default false,
    deleted_at               timestamptz,
    deleted_by_user_id       uuid references users(id),
    constraint chk_attach_size check (size_bytes > 0 and size_bytes <= 50331648)  -- 48 MB
);

create index ix_attach_entity on attachments(entity_type, entity_id) where not is_deleted;
```

## 9. Audit log (partitioned)

```sql
create table audit_log (
    id                  uuid not null default gen_random_uuid(),
    actor_user_id       uuid,                          -- null for system actions
    action              varchar(64) not null,           -- CREATE/UPDATE/DELETE/STATE_TRANSITION/PERMISSION_GRANT/LOGIN/SECURITY_EVENT
    entity_type         varchar(64) not null,
    entity_id           uuid,
    before_json         jsonb,
    after_json          jsonb,
    change_summary_json jsonb,                          -- RFC 6902 JSON Patch
    ip_address          inet,
    user_agent          text,
    trace_id            varchar(64),
    prev_hash           varchar(64),                     -- hash chain pointer
    row_hash            varchar(64) not null,
    at                  timestamptz not null default now()
) partition by range (at);

-- Initial partition
create table audit_log_2026_05 partition of audit_log
    for values from ('2026-05-01') to ('2026-06-01');

create index ix_audit_entity_at on audit_log(entity_type, entity_id, at desc);
create index ix_audit_actor_at on audit_log(actor_user_id, at desc);
create index ix_audit_at on audit_log(at desc);

-- Append-only: prevent UPDATE and DELETE via trigger
create or replace function audit_log_immutable() returns trigger as $$
begin
    raise exception 'audit_log is append-only';
end;
$$ language plpgsql;

create trigger audit_log_no_update before update on audit_log
    for each row execute function audit_log_immutable();
create trigger audit_log_no_delete before delete on audit_log
    for each row execute function audit_log_immutable();
```

Partition rotation: scheduled job creates next month's partition on the 25th of each month. Partitions older than 24 months are detached and archived to compressed CSV in cold storage (see `deployment.md`).

The `row_hash` and `prev_hash` columns form a tamper-evidence chain; a monthly integrity check verifies the chain.

## 11. Form / workflow / data migration discipline

**Three migration folders:**

```
backend/src/main/resources/db/migration/      -- Flyway schema (V001__, V002__...)
backend/src/main/resources/db/data/           -- Flyway data (V001_001__seed_zones.sql)
backend/src/main/resources/db/jsonb-migration/ -- Kotlin classes for breaking schema_json migrations
```

**Naming.** Schema: `V{NNN}__{description}.sql`. Data: `V{NNN}_{NNN}__{description}.sql`. JSONB migrations: Kotlin class `V{YYYY_MM_DD}_{NNN}__{description}.kt`.

**Rules (enforced by CI):**

1. Schema migrations are immutable once merged. CI fails if a previously-merged migration file has been modified. New work always goes into a new file.
2. Adding a field to an existing form requires inserting a new `form_definitions` row (new version), not an `ALTER TABLE`. Exception: if the field needs to be filterable at scale, add a column AND a `form_definitions` version, with a trigger to keep them in sync.
3. Breaking `form_definitions` changes (rename, type-narrow, remove required) must ship with a Kotlin migration class under `db/jsonb-migration/` that transforms each affected `activity_records.data_json` from the old shape to the new. Migration runs on first read of an old record.
4. Removing a column requires two migrations: one to deprecate (rename to `{column}_deprecated`), one in a later release to drop. Allows a rollback window.

## 12. Indexing strategy summary

The indexes shown inline are the minimum set. Plus across the schema:

- Every FK has a btree index (Postgres doesn't auto-index FKs).
- Every JSONB column we filter on gets a GIN index (`gin_{table}_{column}`).
- Partial indexes on `not is_deleted` are used for tables where soft-deletes are common (most domain tables).
- Composite indexes on the hot paths: `(designation_code, primary_zone_id)` on users, `(zone_id, lifecycle_state)` on projects, `(activity_type_code, status)` on project_activities.
- Audit log: `(entity_type, entity_id, at desc)` and `(actor_user_id, at desc)` cover the right-panel history queries and "things I did" queries.

Expression indexes will be added during Phase 2 performance work for specific JSONB paths that show up in dashboard queries (e.g., `((data_json->>'state_code'))` for filtering villages by section state).

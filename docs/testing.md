# PIA Tracker — Testing

**Status:** Draft v1.
**See also:** `architecture.md` § 11 (test architecture); `phasing.md` (per-sub-phase acceptance gates).

This document specifies the test pyramid, naming conventions, the toolchain, and the per-component test obligations.

---

## 1. Pyramid

We aim for a conventional pyramid, weighted slightly more toward integration on the backend (because the database-and-JSONB layer is where bugs hide) and slightly more toward component tests on the frontend (because RJSF and Ant Design have plenty of surface to misuse).

| Layer | Backend (~ratio) | Frontend (~ratio) |
|---|---|---|
| Unit | 50% | 35% |
| Integration / component | 35% | 50% |
| End-to-end (Playwright) | 10% | 10% |
| Property-based / contract / load | 5% | 5% |

Numbers are heuristics, not gates. The gate is the per-sub-phase acceptance criteria in `phasing.md`.

## 2. Backend testing

### Unit tests

JUnit 5, MockK, AssertJ. Run with `./gradlew test`.

Targets:

- Domain services with their dependencies mocked.
- Workflow engine state transitions (every transition + every forbidden actor + every missing requirement).
- Form validators (cross-field business rules).
- Schema diff classifier (pairs of schema versions in / verdict out).
- Permission evaluator (matrix of principal × target × permission).
- `SummaryUpdater` event handlers (event in / summary row out).
- Drawing service: state derivation from approver rows.

Naming: `{ClassName}Test.kt` for the class under test. Test methods read as sentences:

```kotlin
@Test
fun `transition fails when actor lacks required role`() { ... }

@Test
fun `summary recomputes when land record changes hectares`() { ... }
```

### Integration tests

Spring Boot's `@SpringBootTest` plus Testcontainers (real PostgreSQL, real MinIO, real ClamAV via the EICAR test signature).

Targets:

- Repository-level queries (JSONB filtering, GIN index usage via `EXPLAIN`).
- Flyway migrations (apply → query → confirm shape).
- API endpoints end to end through `WebTestClient` (controller → service → repo → DB).
- Workflow transition with real DB rows.
- Attachment upload through to MinIO storage.
- ClamAV scan via the EICAR signature (must be quarantined).
- ETag concurrency (parallel updates from two test users).

Naming: `{Feature}IntegrationTest.kt`.

Testcontainers reuse the same Postgres container across tests via `@Container` `withReuse(true)` — much faster than a fresh DB per test. Schema is reset by Flyway clean+migrate between test classes.

### Property-based tests

jqwik for state-space and shape exploration.

Targets:

- From any state in the workflow, only the configured transitions are reachable.
- Any sequence of approve/send-back operations on a drawing terminates in `APPROVED` or stays in `IN_APPROVAL`/`SENT_BACK`.
- Audit log hash chain integrity holds under arbitrary insertion sequences.
- The schema diff classifier is consistent: `diff(A, B) == "BackwardsCompatible"` implies an instance valid under A is also valid under B.
- JSONB roundtrip: any valid form data survives save → reload → save unchanged.

### Contract tests

Pact between backend and frontend for the OpenAPI surface.

Approach: the OpenAPI spec generated by springdoc is the contract. Frontend's generated client is the consumer. CI runs:

1. Backend builds and emits OpenAPI spec.
2. Frontend generates its client from the spec and compiles.

A breaking change to a controller signature breaks the frontend build. We don't run Pact's broker infrastructure for v1; OpenAPI as the contract is enough.

### Load tests

k6 scripts in `backend/load/`.

Targets:

- Login burst: 100 concurrent dummy-auth selections in 10 seconds.
- Project tree load: 50 concurrent users opening their primary project's tree.
- Form save: 30 concurrent updates to different records, 30 to the same record (ETag conflicts handled).
- Dashboard query: 100 concurrent zone-dashboard loads.

Targets: p95 < the budgets in `architecture.md` § 14. Run nightly in CI (Phase 2 onward); ad-hoc during Phase 1 development.

## 3. Frontend testing

### Unit tests

Vitest. Run with `npm test`.

Targets:

- Pure utility functions (date formatting, chainage parsing, hectares display).
- Hooks in isolation (`useUserPicker`, `useDashboardData`) with mocked TanStack Query.
- Reducers / zustand stores.
- Form validators (mirroring backend validators where applicable — fast feedback for the user).

### Component tests

Testing Library + Vitest, with `jsdom`.

Targets:

- Each archetype's primary container renders correctly with seeded props.
- User Picker filters by context.
- Tree component expands / collapses / selects.
- Record Edit form renders given a form definition + data; submits with the right shape.
- Comments composer with markdown + @mention typeahead.
- Notification bell badge updates on poll.

Naming: `{Component}.test.tsx` co-located with `{Component}.tsx`.

### End-to-end tests

Playwright. Run with `npm run e2e`, against a docker-compose'd test environment.

Targets (the "golden paths" covering every archetype):

1. EDGS/C-I creates a project. CAO/C allocates to a CE/C. CE/C assigns Dy CE/Cs and designates a Nodal. Dy CE/C creates a Land Acquisition activity.
2. Dy CE/C creates a village record, fills SRP section, submits. Nodal verifies. CE/C authenticates.
3. Dy CE/C creates a Drawing Approval record (ESP). Submits. Each approver in turn approves. Drawing ends APPROVED.
4. Send-back path: Nodal sends back. Dy CE/C edits + resubmits. Nodal verifies. CE/C authenticates.
5. Drawing send-back: approver sends back. Dy CE/C re-submits. Approvers re-act.
6. Bulk authenticate: CE/C selects 5 records and authenticates them in one action.
7. Inbox: a Dy CE/C with 3 pending items sees them; clicking one navigates to its edit page.
8. Dashboard: zone-scope dashboard loads with summary tables; clicking through to a project drills down.
9. Excel export: project-scope export downloads a parseable .xlsx with the expected sheets.
10. Project lifecycle: hold → resume → complete.
11. Accessibility: each archetype passes axe-core baseline checks (keyboard navigation, ARIA, contrast).

E2E tests share a fixture project per test suite, isolated by Postgres schema (one schema per Playwright worker). DB state reset between test files.

### Visual regression (Phase 2)

Playwright screenshots of each archetype in light and dark themes; pixel-diff with tolerance against committed baselines. Detects accidental Ant Design upgrades or theme regressions.

### Accessibility tests

axe-core integrated into Playwright. Every archetype test calls `await injectAxe(page); await checkA11y(page)`. CI fails on any new violation (existing violations grandfathered via a known-issues allowlist that shrinks over time).

## 4. Cross-cutting test obligations

For every domain change, the PR must contain:

- **Schema migration**: a Flyway file. CI re-runs migration from `V001` to confirm it applies cleanly.
- **Test data**: if the change is to a form definition, a fixture record file in `tests/fixtures/{form_code}_valid.json` and `_invalid.json`.
- **Audit assertion**: if the change introduces a new state transition or permission grant, an audit log assertion is added confirming the right row is written.
- **Permission test**: if the change introduces a new endpoint or action, a 403 test for the wrong role.
- **Documentation**: if the change crosses an architectural boundary, a doc update in `docs/` referenced by the PR.

## 5. Test data and seeding

A test seeder lives in `backend/src/test/kotlin/in/gov/ir/pia/seed/`. It creates:

- 17 zones, ~50 divisions.
- A full set of designations.
- Seeded users — one per (designation, zone) combination, plus a few cross-zone test users.
- Three test projects in different lifecycle states, with activities and records.

The seeder is idempotent and runs before each integration test class via a `@BeforeAll` hook.

For E2E tests, a more elaborate `e2e-seed.sql` creates a richer fixture: 5 projects across 3 zones, each with multiple activities and records in various workflow states. Drawings with partially-approved checklists. SLA-breached records.

The seed data is intentionally separate from the production seed (zones, designations only; no projects, no records).

## 6. CI pipeline (Phase 2 — Gitea Actions)

Stages, in order, fail-fast:

1. **Lint + format**: ktlint, Detekt, ESLint, Prettier, json-schema check on every `form_definitions` migration.
2. **Type check**: TypeScript `tsc --noEmit`.
3. **Unit tests**: `./gradlew test` (backend) + `npm test` (frontend). Parallel.
4. **Integration tests**: `./gradlew integrationTest` (Testcontainers).
5. **Property tests**: `./gradlew propertyTest`.
6. **Build artifacts**: backend JAR, frontend dist, OpenAPI spec.
7. **Frontend client regen**: `npm run generate-api-client`; CI fails if regenerated client differs from committed (forces explicit acceptance).
8. **Docker image build**: backend and frontend images. Trivy scan; critical/high CVEs fail.
9. **E2E**: docker-compose'd environment; Playwright tests; axe-core; visual regression (Phase 2+).
10. **Load smoke**: k6 short scripts (5 min) confirming budgets met.

Total target: under 25 minutes from push to PR-ready signal.

For Phase 1 (Level 0 — Lefthook + Makefile), only stages 1–4 run locally. CI proper lands in Phase 2.

## 7. Determinism and flakiness

Test rules:

- No `Thread.sleep` — every wait is on a condition (`await element.isVisible()`, `eventually { repo.findById(id).isPresent() }`).
- No randomness in test inputs unless property-based — fixed seeds for any pseudo-random fixture data.
- Timezone fixed to `Asia/Kolkata` in test JVM args; no implicit reliance on the runner's local time.
- Database time queries use a clock abstraction (`Clock`) injected as a bean; tests use a fixed clock.
- No shared mutable state between tests; each test owns its data.

A flaky test is treated as a P1 bug: opens a ticket, gets `@Ignore`d within 24 hours, fixed within a week.

## 8. Coverage

JaCoCo for backend, Vitest's c8 for frontend. Reported in CI but not gated as a percentage.

What we actually gate:

- New code paths must have at least one test exercising the happy path AND one exercising the unhappy path (validation failure, permission denied, ETag stale, etc).
- New endpoints must have at least: 200 happy, 403 wrong role, 404 missing, 409 stale ETag, 422 validation.
- New workflow states must have at least one test per allowed transition and one per forbidden actor.

Reviewer enforces these in PR review. CI provides the coverage report for context.
